1. システム概要

BirdProject は Raspberry Pi で録音した野鳥の鳴き声を、 N97 サーバー上で BirdNET（神経回路網ベースの鳥類音声認識モデル）を使って自動判定し、 結果をこの Web UI で確認・管理するシステムです。

  • 録音は設定したセグメント長（デフォルト 60 秒）単位の WAV ファイルとして Pi 側に保存されます。
  • BirdNET は録音を内部で 5 秒のチャンクに分割して推論します。
  • 1 ファイル内に複数の種が検知された場合、種ごとに 1 レコードとして DB に記録されます。
  • 観察記録として「この時間・この場所でこの種を確認した」という単位で管理できます。
  • ハードウェアとして、3D プリンタで製作するパラボリックマイクを併用可能です（詳細は 文書一覧 参照）。

2. データフロー

🎙 Raspberry Pi 録音（arecord）
→ rsync / 10分ごと
📂 N97 raw_ingest/ WAV を受信
→ cron / 10分ごと
🧠 BirdNET 推論 process.py
→ 仕分け・DB記録
🌿 Web UI 確認・レビュー

各ステップの詳細

ステップ 処理内容 場所
録音 設定した時間帯に 5 分ごとの WAV ファイルを生成。arecord で 48kHz / 16bit / モノラル録音 Pi (record.sh)
転送 10 分ごとに rsync で N97 の raw_ingest/mushipi-bird01/ へ push。転送済みファイルは Pi 側から自動削除 Pi (sync_to_n97.sh)
推論 ステレオをモノラルに変換し BirdNET で推論。5 秒チャンク単位で全種の確信度を取得 N97 (process.py)
仕分け 確信度に応じて detected/ / review/ / unknown/ に振り分け N97 (process.py)
DB 記録 種ごとに 1 レコードを SQLite に INSERT N97 (db.py)

3. ステータスの定義

各レコードには以下の 3 つのステータスがあります。

confirmed 確定

BirdNET の確信度が 70% 以上の検知に自動付与されます。 信頼度が高く、種の同定がほぼ確実と判断されたレコードです。 データエクスポート（学習用データセット作成）の対象になります。

pending レビュー待ち

確信度が 10% 以上 70% 未満の検知に自動付与されます。 種の可能性はあるものの確証が低い状態です。 音声を再生して内容を確認し、「確定」または「ノイズ」に手動で振り分けてください。

操作方法： 一覧の「確定」ボタンを押すと confirmed に、「ノイズ」ボタンを押すと noise に変更されます。

noise ノイズ

確信度が 10% 未満の場合に自動付与されます。 または、pending レコードをレビューして「ノイズ」と判断した際に手動設定します。 風音・雨音・人工音など、野鳥の鳴き声ではないと判断された録音です。

4. 確信度（Confidence）の見方

確信度は BirdNET が「この種である」と判断する確率を 0〜100% で表します。 5 秒チャンクごとに算出され、同じ種が複数チャンクで検知された場合は最高値を採用します。

90%：非常に高い確信度。ほぼ確実にその種
70%：confirmed 閾値。これ以上が自動確定
40%：pending 範囲。要レビュー
10%：noise との境界値
閾値の設定値： CONF_HIGH = 0.7（confirmed 下限）、 CONF_LOW = 0.1（BirdNET API の最低フィルタ値） — process.py の先頭で変更できます。

統計（/stats）

DB 全体の集計データを表示します。

  • ステータス別件数：confirmed / pending / noise の総数と合計
  • 上位検出種（Confirmed）：confirmed レコードの種名ランキング（上位 20 種）
  • 時間帯別検出件数：0〜23 時の検出件数を棒グラフで表示。野鳥の活動時間帯の把握に使えます

7. ファイル保存先の構造

推論後の WAV ファイルは以下の構造で BirdProject/processed/ 以下に保存されます。

BirdProject/processed/
└── 2026-03-18/ # 録音日ごと
    ├── detected/ # confirmed（確信度 70% 以上）
    │   └── Horornis_diphone/ # 科学名（スペース→アンダースコア）
    │       └── 20260318_050012.wav
    ├── review/ # pending（確信度 10〜70%）
    │   └── 20260318_051500_Conf45_Japanese_Bush_Warbler_None.wav
    │                        # 書式: 日時_Conf確信度_Top1種名.wav
    └── unknown/ # noise（確信度 10% 未満）
        └── 20260318_053000.wav
同じ WAV ファイルから複数種が confirmed された場合、物理ファイルは Top1（最高確信度）の科学名ディレクトリに 1 つだけ保存されます。 複数の DB レコードが同じ file_path を参照します。

8. データエクスポート

GET /export/dataset にアクセスすると、 confirmed ステータスのレコードに紐づく WAV ファイルを ZIP 形式でダウンロードできます。

ZIP 内の構造：

bird_dataset.zip
├── Japanese_Bush_Warbler/ # 種名（スペース→アンダースコア）ごとのフォルダ
│   └── 20260318_050012.wav
└── Brown-eared_Bulbul/
    └── 20260318_052300.wav
注意：同じ WAV ファイルが複数種で confirmed されている場合、ZIP 内に重複してコピーされます。

9. FAQ

「今すぐ取り込む」を押しても何も起きない
前回の取り込みが実行中の可能性があります（ロックファイル /tmp/birdproject_ingest.lock が残存）。 または raw_ingest/mushipi-bird01/ に WAV ファイルがない場合はスキップされます。 ingest.log を確認してください。
1 つの WAV から複数のレコードが生成された
正常動作です。5 分の録音内で確信度 70% 以上の種が複数検知された場合、 種ごとに 1 レコード作成されます。すべてのレコードが同じ音声ファイルを参照しているので、 どのレコードのプレーヤーを再生しても同じ録音が聞けます。
設定を保存したら「SSH エラー」のような動作になった
N97 から Pi への SSH 鍵認証が通っていない可能性があります。 また、sudo systemctl restart bird-record.service を mushipi ユーザーがパスワードなしで実行できるように、 Pi 側の /etc/sudoers.d/mushipi-bird が設定されているか確認してください。
Pi が録音時間帯内なのに arecord が動いていない
bird-record.service の起動確認： sudo systemctl status bird-record.service。 USB マイクの認識を確認：arecord -l で hw:0,0 が存在するか確認してください。 接続し直した後に sudo systemctl restart bird-record.service で再起動できます。
sync.log に rsync エラーが出る
N97 の Tailscale が切れているか、Pi 側の SSH 鍵（~/.ssh/id_ed25519_n97）が N97 の authorized_keys に登録されていない可能性があります。 ssh mushipi@100.102.9.77 echo OK で手動接続を確認してください。
確信度の閾値を変えたい
BirdProject/process.py の冒頭にある CONF_HIGH = 0.7（confirmed 下限）と CONF_LOW = 0.1（pending 下限）を変更してください。 変更後は次回の取り込みから適用されます（既存レコードは変更されません）。
セグメント長を短くしたら検知数が減った
短いファイルほど BirdNET の 3 秒分析窓の数が減るため、 鳥が鳴いた瞬間がウィンドウに入らず検知漏れが起きやすくなります。 現在は overlap=1.5 秒のオーバーラップを設定しているため 窓を重ねて分析していますが、30 秒未満では効果が限定的です。 検知数が重要な場合は 60 秒以上を推奨します。
Pi が複数台になったらどうする？
各 Pi ごとに異なる --pi-id（例: mushipi-bird02）を使い、 raw_ingest/mushipi-bird02/ へ push するようにします。 ingest_and_process.sh に Pi ごとの処理ブロックを追加するか、 cron で Pi ごとに別の process.py --pi-id xxx を実行するようにしてください。 Web UI のフィルタは Pi ID に対応済みです。
